# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repo shape

Pantopus is a four-client platform sharing one backend:

- `backend/` — Express 5 + Supabase + Socket.IO (CommonJS, Node 20).
- `frontend/apps/web/` — Next.js 15 App Router, React 19, Tailwind.
- `frontend/apps/ios/` — Swift 5.10 / SwiftUI / `@Observable`, generated by XcodeGen from `project.yml`.
- `frontend/apps/android/` — Kotlin 2.0 / Jetpack Compose / Hilt, Gradle KTS.
- `frontend/packages/*` — TS packages (`@pantopus/api`, `theme`, `types`, `ui-utils`, `utils`) consumed by **web + backend only**.
- `pantopus-seeder/` — Python 3.12 AWS Lambda (SAM) content seeder.
- `supabase/` — `config.toml` + SQL migrations.

The JS/TS side is a **pnpm + Turborepo** monorepo (`pnpm-workspace.yaml` lists `backend`, `frontend/apps/web`, `frontend/packages/*`). The two native apps are **not** workspaces — never add a `package.json` under `frontend/apps/ios/` or `frontend/apps/android/`.

The native apps do **not** consume the shared TS packages. The REST + Socket.IO API is the only contract between iOS/Android and the backend; if a backend response shape changes, the iOS DTOs (`Pantopus/Core/Networking/Models/`) and Android DTOs (`app/src/main/java/app/pantopus/android/data/api/models/`) must be hand-mirrored.

Each platform has its own deeper conventions file — read it before editing that platform:

- `frontend/apps/ios/CLAUDE.md` — MVVM + `@Observable`, endpoint/DTO recipe, shared archetypes (`ListOfRows`, `Wizard`, `Form`, `ContentDetail`), four-state rule, design tokens.
- `frontend/apps/android/CLAUDE.md` — same conventions mirrored for Compose + Hilt + StateFlow.

Both insist on **parity**: same render states, same testIDs (`accessibilityIdentifier` ↔ `testTag` ↔ `data-testid`), same endpoint + query params on iOS/Android/web.

## Backend architecture (the bits you have to know)

**Route mount order matters.** Routers with static paths (e.g. `/api/businesses/search`, `/api/businesses/seats/*`, `/api/businesses/founding-offer/status`, `/api/businesses/:businessId/verify/*`) **must** be mounted in `app.js` before the generic `businessRoutes` whose `/:businessId` would otherwise swallow them. Same pattern for `homeIam` / `homeOwnership` / `homeGuest` before `homeRoutes`, and for `magicTaskRoutes` before `gigRoutes`. When adding a route file, mirror the existing comment markers in `app.js`.

**Webhooks must use raw bodies.** Stripe (`/api/webhooks/stripe`), Lob (`/api/v1/webhooks/lob`), and the internal email-inbound endpoint are mounted with `express.raw(...)` **before** `bodyParser.json()`. Do not move them.

**Feature flags gate whole route families.** `utils/featureFlags.js` exposes `isIdentityFirewallEnabled()`, `isPersonaEnabled()`, `isPersonaBroadcastEnabled()`. The persona routes use a UUID gate at the router level (Express 5 / path-to-regexp 8 dropped the inline `:id(regex)` shorthand). When adding a new persona sub-router, mount under `/api/personas/:id/...` and let the router reject non-UUIDs via `next('router')` so handle-shaped URLs fall through to the public `personaRoutes`.

**Auth pipeline.** `middleware/verifyToken.js` prefers Bearer (mobile) over httpOnly cookie (web), caches `User.role` for 60s (1000-entry LRU), and sets `req._authMethod` which downstream CSRF (`middleware/csrfProtection.js`) keys off. Apply CSRF **after** `verifyToken`, not before.

**Background jobs are in-process.** `jobs/index.js` registers all `node-cron` jobs from `startJobs()` called at the end of `server.listen()` in `app.js`. A new job = a file under `backend/jobs/`, plus a `cron.schedule(...)` line in `jobs/index.js`. There is no separate worker process.

**Supabase access.** Two clients live in `config/`: `supabase.js` (anon, RLS-bound) and `supabaseAdmin.js` (service role, bypasses RLS). Always use `supabaseAdmin` for writes from server-side business logic; the anon client is for reads where RLS should apply.

**Boot guard.** `assertPersonaFollowViewActive()` runs before `server.listen()` and refuses to boot when `PERSONA_FOLLOW_VIEW_ACTIVE=false` — this catches the "SQL rolled back but code wasn't" deploy mismatch. Don't bypass it; fix the underlying mismatch.

**Push notifications** still use `expo-server-sdk` (legacy from a removed RN app). The native apps will need APNs (iOS) / FCM (Android) migration in `services/pushService.js`; the device-token schema needs a `platform` column when that lands.

## Identity Firewall + privacy gates (do not regress)

The Identity Firewall enforces that personal identity never leaks across persona contexts. Several CI guards exist and run via `pnpm test:privacy` (see `backend/scripts/ci/run-privacy-gates.js`):

1. Serializer forbidden-keys contract (`tests/unit/privacy/serializerForbiddenKeys.test.js`).
2. Notification template + cross-context harness.
3. Legacy identity-aliases scan (`check-legacy-identity-aliases.js`).
4. Legacy UI terms scan (`check-legacy-ui-terms.js`).
5. Creator-select guard (`check-creator-select.js`).
6. Raw personal-identity SELECT guard (`check-raw-personal-selects.js`).
7. Nested User-select guard (`check-nested-user-selects.js`).
8. Audience-profile end-to-end integration.

If you add a new serializer, new notification template, or new User select, the guards will tell you which contract you crossed. Run `pnpm test:privacy` from `backend/` before pushing changes that touch identity, personas, or notifications.

## Native CI guards (raw hex / icon enums)

Both native apps fail CI if feature code uses raw colour or icon values:

- iOS: `make lint` runs `verify-icons` which forbids `Image(systemName:)` and Lucide names outside `PantopusIcon`. `Features/**` may not contain hex literals; tokens only (`Theme.Color.*`, `Spacing.*`, `Radii.*`).
- Android: `./gradlew verifyPantopusIcons` enforces `PantopusIconImage(...)`. The Android CI workflow greps `app/src/main/java/.../ui/screens/` for `#RRGGBB(AA)` literals; only `ui/theme/` is exempt. Use `PantopusColors.*` or `PantopusTheme.tokens.*`.

When you re-skin a screen, the design tokens live in `@pantopus/theme` (web), `Theme.Color.*` / `Spacing.*` (iOS), and `PantopusColors.*` / `PantopusTheme.tokens.*` (Android). Add new tokens in all three places together.

## Testing layout

- **Backend Jest** (`backend/jest.config.js`): mocks Supabase, logger, push, notification, and the `verifyToken` middleware via `tests/__mocks__/`. The default runner deliberately includes a handful of integration-shaped tests (audience profile, notifications context, identityCenter, persona payments) because they only need the mocked admin client. Add new integration-shaped tests under `tests/integration/` and the runner will skip them unless they're whitelisted in `testPathIgnorePatterns`.
- **Backend integration** (`backend/jest.integration.config.js`): no mocks, real Supabase, `maxWorkers: 1` because tests share the DB. Set `SUPABASE_URL`, `SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`.
- **Web Jest** lives under `frontend/apps/web/tests/**/*.test.{ts,tsx}` with `@pantopus/api` mocked. Playwright specs (`*.spec.ts`) are excluded from Jest and run via `playwright test` against `localhost:3000`.
- **iOS** tests under `PantopusTests/Features/<Feature>/`. Networking-bound tests drive `APIClient(retryPolicy: .none, session: SequencedURLProtocol.makeSession())` with scripted `SequencedURLProtocol.sequence`.
- **Android** unit tests under `app/src/test/java/.../<feature>/` (JVM, no Robolectric); mockk + Turbine for Flow collection. Paparazzi locks the visual contract — re-record with `./gradlew paparazziRecord`.

## Conventions

- **pnpm only** for JS/TS. `npm install` will mangle the lockfile.
- **CommonJS** on the backend (`require(...)`); ES modules in `frontend/packages/*` and the web app.
- **Snake_case → camelCase** is **not automatic** on iOS REST DTOs — declare explicit `CodingKeys`. It **is** automatic on iOS socket payloads. On Android, both REST and socket use explicit `@Json(name = "...")`. See the per-platform CLAUDE.md for the full rules.
- **No bespoke list/wizard/form/detail shells.** Reuse `ListOfRows`, `Wizard`, `Form`, `ContentDetail` archetypes per platform unless the design genuinely can't fit a slot.
- Each fetchable screen ships **four states**: loading skeleton (not a spinner), empty (with CTA), loaded, error (with Retry). Wrap with the platform's offline banner.
- Don't commit generated artifacts: `Pantopus.xcodeproj/`, `DerivedData/`, `.gradle/`, `build/`, `.next/`, `.turbo/` are gitignored.
